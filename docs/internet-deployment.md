# Internet deployment

Galaxy OS must never publish `next dev`, NestJS port 3001, PostgreSQL, Redis,
or Mailpit. Build first, run Next's installed production server on
`127.0.0.1:3000`, and run NestJS on `127.0.0.1:3001`. Browsers use only the
same-origin `/api/v1` route; Next forwards its closed API allowlist internally.

## Production environment and processes

Create ignored `.env.production` with production secrets held outside Git:

```env
NODE_ENV=production
ALLOW_DEV_AUTH=false
APP_PUBLIC_ORIGIN=https://os.example.com
TRUSTED_PUBLIC_HOSTS=os.example.com
NEXT_PUBLIC_API_URL=/api/v1
INTERNAL_API_URL=http://127.0.0.1:3001/api/v1
TRUST_PROXY=true
```

Also set the database, session, SMTP, sender, and storage values described in
`.env.example`. `APP_PUBLIC_ORIGIN` is the sole base for password-reset and
other email links. It must be a path-free HTTPS origin. Do not use a request
Host header to build links. `TRUST_PROXY=true` is appropriate only with the
loopback bindings and one of the approved local proxy paths below; forwarded
headers are otherwise ignored.

```bash
pnpm install --frozen-lockfile
pnpm db:generate
pnpm --filter @galaxy/api prisma:deploy
pnpm build:api
pnpm build:web
```

The supported foreground commands are `pnpm start:api` and
`pnpm start:web`. `pnpm prod:health` checks both loopback health endpoints.
For persistence, copy the three examples from `infra/systemd` to
`~/.config/systemd/user`, replace `<REPOSITORY_PATH>` and `<PNPM_PATH>` with
absolute non-secret paths, then run:

```bash
systemctl --user daemon-reload
systemctl --user enable --now galaxy-os.target
pnpm prod:health
systemctl --user status galaxy-os-api galaxy-os-web
```

The target starts PostgreSQL and Redis before the API, then the web service;
both application processes restart on failure and receive SIGTERM on
`pnpm prod:stop`. Use `journalctl --user -u galaxy-os-api -u galaxy-os-web`
for logs; authentication bodies and secrets are redacted, but environment
files must never be printed. WSL and Docker Desktop must be running. If the
host does not start WSL automatically, use a Windows startup task that starts
the distribution, not another application process manager.

## Recommended: Cloudflare Tunnel

Architecture: Internet HTTPS hostname → outbound Cloudflare Tunnel →
`127.0.0.1:3000` → same-origin Next API forwarder → `127.0.0.1:3001`.
Dynamic public-IP changes do not require DNS updates or inbound router rules.

1. Remove the router DMZ, the TCP 3000 mapping, Galaxy OS UPnP mappings, and
   every API/database/UDP mapping.
2. In Cloudflare Zero Trust, create a tunnel and install `cloudflared` using
   the exact command generated by Cloudflare. Do not paste that command or its
   token into source, shell history, tickets, or logs.
3. Create a **Published Application** route with the chosen application
   subdomain and service `http://127.0.0.1:3000`.
4. For locally managed configuration, copy
   `infra/cloudflare/config.yml.example` outside Git, replace the hostname and
   tunnel placeholders, restrict credential-file permissions, and run
   `cloudflared tunnel ingress validate` before installing the service.
5. Confirm `cloudflared` is healthy and restart-enabled with the platform
   service manager. Logs may include tunnel IDs and request metadata but must
   never include tokens or credential JSON.
6. Verify the public hostname over HTTPS, Login/logout, `/api/v1/me`, and the
   external route's 404 fallback. Only then consider enabling HSTS at the edge.

No router Virtual Server rule, DMZ, inbound firewall exception, DDNS updater,
or stable public IPv4 is required. Roll back by disabling the published route
and tunnel service; local loopback services remain private.

## Alternative: DDNS and Caddy

Architecture: Internet HTTPS domain → router TCP 80/443 → Caddy →
`127.0.0.1:3000`. Copy `infra/caddy/Caddyfile.example` outside Git, set
`APP_DOMAIN` in Caddy's protected service environment, validate the file, and
restart Caddy. Caddy obtains and renews certificates and redirects HTTP to
HTTPS automatically when public DNS resolves to the router, TCP 80 and 443
reach Caddy, and the domain is publicly eligible for certificate issuance.

Create an A record and run one DDNS updater on the always-on host. It must:

- discover the current public IPv4, compare it with the existing record, and
  update only after a real change at a reasonable interval such as 5 minutes;
- use an API token scoped only to DNS edit for the selected zone, never a
  Global API key, and store the token in a protected file outside Git;
- make idempotent updates and log success/failure without request headers,
  tokens, or provider response bodies that may contain secrets.

Router rules are only TCP external 80 → the Windows LAN address port 80 and
TCP external 443 → that address port 443. Do not forward UDP, 3000, 3001,
5432, 6379, or 8025; disable DMZ and relevant UPnP. In Administrator
PowerShell allow only Caddy's public ports:

```powershell
New-NetFirewallRule -DisplayName "Galaxy OS HTTPS" -Direction Inbound -Action Allow -Protocol TCP -LocalPort 80,443 -Profile Public,Private
```

If Caddy runs inside NAT-mode WSL, add Windows portproxy rules for TCP 80 and
443 to the current WSL IPv4 and update them whenever `wsl hostname -I`
changes. Prefer mirrored networking or running Caddy on Windows where
available. Before direct forwarding, compare the router WAN IPv4 with an
independently observed public IPv4. A mismatch indicates carrier-grade or
double NAT; use Cloudflare Tunnel or ask the ISP for routable service.

Rollback by removing router 80/443 rules, the matching Windows firewall and
portproxy rules, the DDNS updater, and the Caddy service. Remove the DNS record
when it is no longer used.

## Temporary public-IP diagnostic only

This is not production-ready. Never enter a password over it.

Temporarily map router TCP external 3000 to the Windows LAN address port 3000,
then in Administrator PowerShell point Windows at the current WSL address:

```powershell
netsh interface portproxy add v4tov4 listenaddress=0.0.0.0 listenport=3000 connectaddress=<CURRENT_WSL_IP> connectport=3000
New-NetFirewallRule -DisplayName "Galaxy OS TEMP 3000" -Direction Inbound -Action Allow -Protocol TCP -LocalPort 3000 -Profile Public,Private
```

Use only a health-page request to diagnose reachability. The public IP and WSL
IP may change. Immediately remove the router rule and run:

```powershell
netsh interface portproxy delete v4tov4 listenaddress=0.0.0.0 listenport=3000
Remove-NetFirewallRule -DisplayName "Galaxy OS TEMP 3000"
```

Public HTTP on `http://<public-ip>:3000` is never safe for Login and must not
be described or treated as a deployment mode.
